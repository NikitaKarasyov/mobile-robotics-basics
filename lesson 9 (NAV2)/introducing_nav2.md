# Знакомство с Nav2

## Цель туториала
Понять, из чего состоит навигационный стек **Nav2**: какие ноды он запускает, как они связаны топиками и какие tf-фреймы обязательны для работы навигации. После этой статьи вы будете знать, «кто за что отвечает», прежде чем запускать навигацию в симуляции.

---

## Что такое Nav2?
**Nav2 (Navigation2)** — это навигационный стек ROS 2. Он решает главную задачу мобильного робота: **доехать из точки A в точку B безопасно и без столкновений**.

Если разложить задачу на шаги, Nav2:

1. принимает целевую позу (точку назначения и ориентацию);
2. строит глобальный маршрут по карте;
3. ведёт робота вдоль маршрута, публикуя команды скорости в топик `cmd_vel`;
4. объезжает препятствия, которые видят сенсоры (например, лидар);
5. выполняет восстановление, если робот «застрял» (отъехать назад, покрутиться на месте).

Это та же связка «подписка на сенсоры → вычисления → публикация команд», что и в предыдущих уроках, только собранная из готовых, настраиваемых нод.

## Архитектура: основные серверы Nav2
Nav2 устроен модульно. Каждая задача — отдельная нода-сервер, а координирует их **дерево поведения** (behavior tree):

| Нода | Задача |
|---|---|
| `bt_navigator` | Координатор. Выполняет дерево поведения: «построй маршрут → следуй по нему → при сбое выполни восстановление» |
| `planner_server` | **Глобальный планировщик**: строит маршрут от текущей позиции до цели по карте |
| `controller_server` | **Контроллер** (локальный планировщик): следует вдоль маршрута и публикует `cmd_vel`, учитывая препятствия рядом с роботом |
| `behavior_server` | Поведения восстановления: backup, spin, wait |
| `smoother_server` | Сглаживание построенного маршрута |
| `map_server` | Загружает и публикует готовую карту (топик `map`) |
| `amcl` | Локализация: определяет, где робот находится на карте |
| `lifecycle_manager` | Управляет запуском и остановкой всех нод стека |

Важно: серверы Nav2 — это **action-серверы** (см. [урок 3](<../lesson 3 (actions and services)/understanding_actions.md>)). Например, цель навигации отправляется как действие `NavigateToPose`: долгая задача с обратной связью и возможностью отмены — ровно тот случай, для которого действия и придуманы.

## Обязательные tf-фреймы
Nav2 не заработает без правильного дерева трансформаций (см. [урок 7, tf2](<../lesson 7 (TF2)/introducing_tf2.md>)). Стандарт [REP-105](https://www.ros.org/reps/rep-0105.html) требует цепочку:

```
map -> odom -> base_link
```

| Фрейм | Что это | Кто публикует трансформацию |
|---|---|---|
| `map` | Неподвижная мировая система координат, привязанная к карте | `amcl` публикует `map -> odom` |
| `odom` | Фрейм одометрии: гладкий, но «дрейфует» со временем | нода одометрии робота публикует `odom -> base_link` |
| `base_link` | Фрейм, жёстко привязанный к корпусу робота | `robot_state_publisher` публикует `base_link -> сенсоры` (из URDF) |

Зачем два «родителя» у робота? Одометрия (`odom -> base_link`) обновляется часто и без скачков — по ней удобно управлять движением. Но она накапливает ошибку. Локализация (`map -> odom`) исправляет этот дрейф, «подтягивая» фрейм одометрии к карте.

Помимо tf, Nav2 нужны: топик с данными лидара (`scan`), карта (`map`) и робот, который слушает `cmd_vel`.

## Lifecycle-ноды
Все серверы Nav2 — это **управляемые (lifecycle) ноды**. В отличие от обычных нод, они проходят явные состояния:

```
unconfigured -> inactive -> active
```

* **unconfigured** — нода запущена, но ничего не делает;
* **inactive** — параметры загружены, ресурсы выделены, но нода не публикует и не обрабатывает данные;
* **active** — нода полноценно работает.

Это нужно для детерминированного старта: `lifecycle_manager` сначала конфигурирует все ноды стека, и только когда все готовы — активирует их в правильном порядке. Если бы ноды стартовали как попало, контроллер мог бы начать вести робота до того, как загрузилась карта.

Посмотреть состояние lifecycle-ноды можно командами:

```bash
ros2 lifecycle nodes          # список lifecycle-нод
ros2 lifecycle get /amcl      # текущее состояние ноды amcl
```

## Установка Nav2

```bash
sudo apt install ros-$ROS_DISTRO-navigation2 ros-$ROS_DISTRO-nav2-bringup
```

* `navigation2` — метапакет со всеми серверами Nav2;
* `nav2_bringup` — готовые launch-файлы и примеры конфигураций.

---

## Контрольные проверки
Убедитесь, что пакеты установлены:

```bash
ros2 pkg list | grep nav2
```

В выводе должны присутствовать, среди прочих: `nav2_bringup`, `nav2_bt_navigator`, `nav2_planner`, `nav2_controller`, `nav2_map_server`, `nav2_amcl`, `nav2_lifecycle_manager`.

Проверьте, что доступен интерфейс действия навигации:

```bash
ros2 interface show nav2_msgs/action/NavigateToPose
```

В ответе вы увидите три секции, разделённые `---`: цель (`pose`), результат и обратная связь — структура действия из урока 3.

---

## Итог
* Nav2 — модульный навигационный стек: планировщик строит маршрут, контроллер ведёт робота, behavior-сервер выполняет восстановление, а bt_navigator всем управляет.
* Для работы обязательна цепочка фреймов `map -> odom -> base_link`: одометрия даёт гладкое движение, локализация исправляет дрейф.
* Серверы Nav2 — lifecycle-ноды, их стартом управляет `lifecycle_manager`.
